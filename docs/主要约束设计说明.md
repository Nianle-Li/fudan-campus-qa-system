# 主要约束设计说明

本文档概述数据库设计中必须在数据库层实施的约束策略，说明约束的类型、原因及在业务场景中的作用，便于开发和运维一致实现与审计。

1. 主键与自增

- 大多数实体采用代理主键（`SERIAL`/`INT AUTO_INCREMENT`），保证单列主键、查询性能与外键引用简单。
- 课程主表使用业务主键 `course_master_code`，确保课程定义层的语义唯一性。

2. 外键与级联策略

- 一律使用 `ON UPDATE CASCADE`，保证主键更新（极少发生）时子表同步。
- 删除策略按语义设置：
  - 强约束关系使用 `ON DELETE RESTRICT`（如建筑→校区、设施→建筑、课程班次→课程），避免误删上位实体导致大量丢失。
  - 事务性或从属关系使用 `ON DELETE CASCADE`（如 course_offering_schedule、course_offering_teacher、user_activity）以便删除上层实例时自动清理相关从属记录。

3. 非空与默认值

- 关键显示字段（姓名、课程名称、学期等）使用 `NOT NULL`，并辅以 `CHECK (TRIM(...) <> '')` 防止空字符串。
- 业务上允许暂缺但不希望为 NULL 的字段使用默认值（如 department、major、open_time、title、organizer 等），便于统计与显示。

4. 唯一约束

- 在需要业务唯一性的场景设置 `UNIQUE`：
  - 校区名全局唯一（`UNIQUE (name)`）。
  - 建筑在同一校区内唯一（`UNIQUE (campus_id, name)`）。
  - 设施在同一建筑内唯一（`UNIQUE (building_id, name)`）。
  - 活动在相同时间与地点的同名冲突通过 `UNIQUE (name, start_time, facility_id)` 控制。

5. 检查约束（CHECK）

- 对枚举类字段采用 `CHECK (col IN (...))`，例如建筑/设施类型、上课周次类型、周内上课日、查询类别、用户活动状态等。
- 对时间逻辑使用 `CHECK (end_time IS NULL OR end_time >= start_time)`，防止结束时间早于开始时间。

6. 复合主键与原子性

- 对于天然的复合标识（如课程开设时段：`(offering_id, day_of_week, start_period, end_period, week_type)`），采用复合主键以保证时段唯一性并利于业务去重。

7. 索引建议

- 索引实现集中在 `migrations/070_create_indexes.sql`，与建表脚本一同由 `db/schema.sql` 执行。
- 为常用外键 JOIN 和关联查询建立索引：
  - `building(campus_id)`、`facility(building_id)`、`course_section(course_master_code)`、`course_offering(course_code)`。
  - `course_offering_schedule(facility_id, day_of_week, start_period, end_period, week_type)`，支持教室/设施占用查询。
  - `course_offering_teacher(teacher_id, offering_id)`，支持“查询某位教师授课列表”。
  - `activity(facility_id, start_time)`，支持按地点和时间查询活动。
- 为统计和排序场景建立索引：
  - `query_log(user_id, query_time DESC)`，支持用户历史查询。
  - `query_log(query_category, query_time DESC)`，支持热门查询类别统计。
  - `user_activity(activity_id, status)`，支持热门活动和参与状态统计。
  - `activity(start_time)`、`course_offering(semester)`、`building(type)`、`facility(type)`，支持近期活动、学期课程和分类统计。
- 对于关键词检索，采用 Postgres `pg_trgm` + GIN 索引支持名称/内容模糊搜索，并为用户、课程、设施、活动、查询记录保留 `tsvector` 生成列，便于后续接入更完整的搜索能力。

8. 文档与迁移一致性

- 所有约束应通过迁移脚本显式声明，避免仅靠应用层校验导致的数据不一致。
- 在无法直接用 CHECK 表达的复杂业务规则（如跨表的一致性）时，应考虑触发器或在事务中以约束性检查实现。

9. 备份与演进策略

- 在修改约束或删除字段前，先执行兼容性检查与迁移脚本，并在低峰期按步骤演进（新增允许 NULL 字段 → 批量填充 → 设 `NOT NULL` 并添加 CHECK）。
