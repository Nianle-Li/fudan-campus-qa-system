## 1. 背景与价值 (Context)

### 1.1 用户故事 (User Story)

- 作为**校园用户**（学生、教职工），希望能够在一个统一的平台快速、准确地查询校园内的课程、建筑、设施及活动等信息，以提高信息获取效率，为学习、工作和生活提供便利。
- 作为**系统管理员**，希望能够方便地对校园基础数据进行集中维护与管理（增、删、改、查），以保证信息的一致性、准确性和时效性。

### 1.2 业务目标 (Business Objective)

建立一个以关系型数据库为核心的校园信息管理与查询系统。该系统旨在实现多源异构信息的统一建模、集中存储与高效查询，为未来可能的智能问答、数据分析等上层应用提供稳定、可靠、结构化的数据基础。

## 2. 需求范围 (Scope)

### 2.1 包含功能 (In-Scope)

- **基础信息查询**：支持通过关键词搜索或自然语言提问的方式，查询校园建筑、设施、课程、教师及活动等核心信息。
- **用户信息查询**：用户可以查看自己的账户信息。
- **多维度关联查询**：
  - 支持基于教师或校区的关联查询。
  - 支持多级关联查询，例如：“查询某栋建筑中的设施信息”。
  - 支持聚合与分组查询，例如：“统计每个校区各类建筑的数量”。
  - 支持课程与教师关联查询，例如：“查询某位教师所教授的课程列表”。
- **用户查询记录管理**：系统记录用户的历史查询，方便用户回顾。
- **用户活动交互记录**：系统可记录用户对活动的交互关联（如“订阅、已报名”等），用于支持热门活动等统计分析。
- **后台数据管理**：授权管理员对所有基础数据表进行增、删、改、查（CRUD）操作。
- **演示与身份假设**：可假设用户已登录；演示时应支持用户身份切换（例如通过前端界面或 API 请求方式）。
- **基础统计分析**：提供简单的统计功能，如建筑总数、课程总数、热门查询词等。

### 2.2 不包含功能 (Out-of-Scope)

- 复杂的用户权限体系及多级角色访问控制。
- 基于用户行为的实时推荐或个性化内容算法。
- 高级自然语言处理（NLP）模型的实现与集成。
- 前端界面的视觉设计、UI优化及复杂的交互逻辑。

## 3. 系统逻辑与数据驱动流程

本系统核心逻辑以数据流转与实体关系为主。前端与后端的实现用于支撑数据库设计的展示与验证。

### 3.1 数据驱动流程

1.  **数据注入**：管理员通过后台管理功能，录入或批量导入校区、建筑、课程、教师、设施、活动等基础数据。
2.  **数据持久化**：数据被存储于关系型数据库中。通过主键、外键等约束保证实体间的关联关系和数据完整性。
3.  **用户请求**：用户通过查询接口发起查询请求。
4.  **请求解析**：系统后台将用户请求解析为结构化的SQL查询语句。
5.  **数据检索**：数据库执行SQL语句，进行单表或多表连接查询，并返回结果集。
6.  **结果呈现**：系统将查询结果格式化后返回给用户。
7.  **行为记录**：系统异步记录用户的查询行为（查询内容、查询类别、时间等），用于后续统计分析。

### 3.2 数据访问模式

- **读操作 (Read)**：系统核心操作，以高频的单表及多表连接（JOIN）查询为主。
- **写操作 (Write)**: 以低频、高权限的后台数据插入（INSERT）、更新（UPDATE）与删除（DELETE）操作为主，且操作受严格的数据库约束（如外键、非空）限制。
- **统计操作 (Analytics)**：以中低频的聚合（Aggregation）和分组（GROUP BY）查询为主。

## 4. 业务规则与数据需求

### 4.1 核心实体关系规则 (Entity Relationship Rules)

为确保数据模型的准确性与扩展性，核心实体间关系定义如下：

- **校区与建筑 (1:N)**：一个校区可以包含多个建筑，每个建筑必须属于唯一一个校区。
- **建筑与设施 (1:N)**：一个建筑可包含多个设施，每个设施必须归属于唯一一个建筑。
- **用户与教师/学生子类 (1:1)**：用户是一个上位实体（超类）。教师与学生是用户的两个子类实体（子类表），通过共享主键方式与 `User(user_id)` 进行 1:1 关联（即子类表的主键同时也是外键）。
- **用户与查询记录 (1:N)**：一个用户可产生多条查询记录，每条记录对应唯一一个用户。
- **活动与设施 (N:1)**：一个活动在一个具体设施（如某教室/会议室/自习室）内举办，一个设施可以承载多个活动。由于每个设施都从属于唯一一个建筑（见“建筑与设施”关系），因此活动记录可通过设施间接映射到对应建筑，从而支持“按建筑查看活动”等查询场景。
- **课程本体与课程班次 (1:N)**：系统将课程稳定信息与课程代码拆分建模。`Course(course_master_code)` 表示课程本体，`CourseSection(offering_code)` 表示课程代码；一个课程主代码可以对应多个课程代码，而每个课程代码只对应唯一一个课程主代码。
- **课程班次与课程开设 (1:N)**：同一个课程代码可以在不同学期重复使用，也可能在同一学期出现并行开设。因此通过 `CourseOffering(offering_id, offering_code, semester)` 表示某个课程代码在某学期的一次实际开设；一个课程代码可以对应多次课程开设。
- **课程开设与授课教师 (M:N)**：同一次课程开设可能由多位教师共同承担，一位教师也可能参与多次课程开设。因此通过独立关系表 `CourseOfferingTeacher(offering_id, teacher_id)` 表示课程开设与教师之间的对应关系，而不是将教师直接固化在课程定义层或课程代码层。
- **课程开设与上课时段 (1:N)**：一次课程开设可以对应多个上课时段，例如一周上两次课或单双周不同安排。因此通过 `CourseOfferingSchedule(offering_id, day_of_week, start_period, end_period, week_type, facility_id)` 记录每次开设的具体时段安排。
- **课程开设时段与设施 (N:1)**：每个课程开设时段在一个具体设施内进行，而一个设施可以承载多个课程开设时段。因此设施归属绑定在课程开设时段层，而不是课程层或课程开设层，以支持“同一门课不同时间段使用不同设施”的场景。
- **用户与活动 (M:N)**：一个用户可以报名参加多个活动，一个活动也可以被多个用户参加。此关系通过一个独立的中间表（用户活动参与表）实现。

### 4.2 数据完整性规则 (Data Integrity Rules)

- **实体完整性**: 每个实体表必须有唯一的主键（Primary Key）来标识每一条记录。
- **参照完整性**: 外键（Foreign Key）引用的值必须在被引用的主键列中真实存在，或为NULL（如果允许）。禁止删除仍被引用的父表记录。
- **域完整性**: 列的数据类型、长度、格式、范围及非空（NOT NULL）约束必须被严格遵守。

## 5. 验收标准 (Acceptance Criteria)

### 5.1 功能验收

- **数据操作**: 所有实体表的增、删、改、查操作均能正确执行，并符合预期。
- **查询场景**:
  - 能够正确执行基础的单实体查询。
  - 能够正确执行“查询某个校区的建筑列表”。
  - 能够正确执行“查询某栋建筑中的设施信息”。
  - 能够正确执行“查询某门课程的基本信息”。
  - 能够正确执行“查询某位教师所教授的课程”。
  - 能够正确执行“查询近期校园活动信息”。
  - 能够正确统计并返回“不同校区的建筑数量”。
  - 能够正确统计并返回“每个院系开设的课程数量”。
  - 能够正确统计并返回“最常被查询的校园信息类别”（基于查询记录的查询类别字段）。
  - 能够正确统计并返回“热门活动”（例如按用户活动关联记录数进行排序）。

### 5.2 数据正确性验收

- **约束有效性**: 主键、外键、唯一、非空等约束全部生效，非法操作会被数据库拒绝。
- **数据规范性**: 数据库设计满足第三范式（3NF），不存在明显的非主属性对码的部分/传递函数依赖，数据冗余度低。
- **无异常数据**: 不存在因更新、删除操作导致的孤立数据或数据不一致现象。
