## 一、关系模式初稿

根据题目中的实体、联系及基数约束，可得到如下关系模式初稿：

1. **用户**（**用户编号**，姓名，院系）
2. **教师**（**用户编号**，职称）
3. **学生**（**用户编号**，专业）
4. **校区**（**校区编号**，校区名称，地址）
5. **建筑**（**建筑编号**，建筑名称，建筑类型，校区编号）
6. **设施**（**设施编号**，设施名称，设施类型，开放时间，建筑编号）
7. **课程**（**课程编号**，课程名称，上课时间，设施编号）
8. **教师授课**（**教师编号**，**课程编号**，**开课学期**）
9. **活动**（**活动编号**，活动名称，开始时间，结束时间，主办单位，建筑编号）
10. **查询记录**（**记录编号**，查询类别，查询内容，查询时间，用户编号）
11. **用户活动参与**（**用户编号**，**活动编号**）

其中：

- **主要实体表**：用户、校区、建筑、设施、课程、活动、查询记录
- **子类扩展表**：教师、学生
- **关系表**：教师授课、用户活动参与

## 二、主键与外键设计说明

### 1. 主键设计

- 用户表主键：**用户编号**
- 教师表主键：**用户编号**
- 学生表主键：**用户编号**
- 校区表主键：**校区编号**
- 建筑表主键：**建筑编号**
- 设施表主键：**设施编号**
- 课程表主键：**课程编号**
- 教师授课表主键：**（教师编号，课程编号，开课学期）**
- 活动表主键：**活动编号**
- 查询记录表主键：**记录编号**
- 用户活动参与表主键：**（用户编号，活动编号）**

说明：

- 教师、学生是用户的子类实体，因此采用“**共享主键**”方式，子表主键直接继承父表主键“用户编号”。
- 教师授课表用于表示“教师—教授—课程”这一联系。由于该联系具有属性“开课学期”，因此采用**联合主键**（教师编号，课程编号，开课学期）来唯一标识某教师在某学期开设某课程这一事实。
- 用户活动参与表用于实现“用户—参加—活动”的多对多联系，因此采用**联合主键**（用户编号，活动编号）来唯一标识一次参与关系。

------

### 2. 外键设计

- 教师.用户编号 → 用户.用户编号
- 学生.用户编号 → 用户.用户编号
- 建筑.校区编号 → 校区.校区编号
- 设施.建筑编号 → 建筑.建筑编号
- 课程.设施编号 → 设施.设施编号
- 教师授课.教师编号 → 教师.用户编号
- 教师授课.课程编号 → 课程.课程编号
- 活动.建筑编号 → 建筑.建筑编号
- 查询记录.用户编号 → 用户.用户编号
- 用户活动参与.用户编号 → 用户.用户编号
- 用户活动参与.活动编号 → 活动.活动编号

------

### 3. 关键字段设置说明

#### （1）教师、学生子类设计

教师表和学生表中的“用户编号”既是各自表的主键，也是指向用户表的外键，用于表示“教师/学生都是用户”的泛化—特化关系。

#### （2）一对多联系处理

对于以下 1:N 联系，不单独建立关系表，而是在 N 端表中设置外键：

- 校区 — 建筑：在建筑表中设置校区编号
- 建筑 — 设施：在设施表中设置建筑编号
- 用户 — 查询记录：在查询记录表中设置用户编号
- 建筑 — 活动：在活动表中设置建筑编号

#### （3）带属性的一对多联系处理

教师与课程之间原本可按 1:N 联系处理，即在课程表中设置教师编号外键；但考虑到“教授”这一联系本身具有“开课学期”属性，因此将其从简单外键形式中独立出来，建立**教师授课表**。

该设计用于表示“某教师在某学期开设某课程”这一事实，能够更清晰地反映关系本身的信息，也体现了当联系具有自身属性时，即使是 1:N 联系，也可以通过单独关系表进行建模。

#### （4）多对多联系处理

用户与活动之间为 M:N 联系，因此单独建立**用户活动参与表**，以保存用户参加活动的对应关系。

#### （5）一对一联系处理

课程与设施之间为 1:1 联系，因此在课程表中设置“设施编号”作为外键，以表示每门课程对应一个设施。
为严格体现 1:1 约束，**课程表中的设施编号应设置唯一约束**。

## 三、各表设计依据

### 1. 用户表

**用户**是系统中的基础核心实体，用于存储所有用户共有的基本信息，包括用户编号、姓名、院系。
由于教师和学生都属于用户，因此将公共属性提取到用户表中，避免在教师表和学生表中重复存储姓名、院系等信息。

------

### 2. 教师表

**教师**是用户的子类实体。教师除具有用户共有属性外，还具有“职称”这一特有属性，因此单独建立教师表。
该表采用与用户表相同的主键“用户编号”，既能体现继承关系，也能避免在用户表中增加大量对非教师无意义的空值字段。

------

### 3. 学生表

**学生**也是用户的子类实体。学生除具有用户共有属性外，还具有“专业”这一特有属性，因此单独建立学生表。
该表同样采用共享主键方式，与用户表形成一一对应关系，体现用户的特化结构。

------

### 4. 校区表

**校区**是独立实体，用于描述学校校区的基本信息，包括校区编号、校区名称和地址。
由于一个校区可以包含多个建筑，因此校区表作为上层实体单独设计，其主键“校区编号”被建筑表引用。

------

### 5. 建筑表

**建筑**用于描述具体建筑物的信息，包括建筑编号、建筑名称和建筑类型。
根据“校区—包含—建筑”为 1:N 联系，在建筑表中增加“校区编号”外键，用于表示每栋建筑属于哪个校区。
同时，建筑还与设施、活动存在联系，因此建筑表也是连接空间资源信息的重要实体表。

------

### 6. 设施表

**设施**用于存储具体设施的信息，包括设施编号、设施名称、设施类型和开放时间。
根据“建筑—包含—设施”为 1:N 联系，在设施表中设置“建筑编号”外键，表示每个设施从属于某一栋建筑。
这样设计能够清晰反映设施的空间归属关系，也便于后续按建筑查询设施信息。

------

### 7. 课程表

**课程**是独立业务实体，用于保存课程编号、课程名称和上课时间等信息。
根据“课程—发生—设施”为 1:1 联系，在课程表中设置“设施编号”外键，表示每门课程对应一个设施。采用将外键放在课程表中的方式，能够直接体现课程的授课地点信息，查询课程安排时也较为方便。

同时，本设计将课程的上课时间和授课地点视为课程层面的相对稳定属性，不进一步细分到不同学期，以保证关系模式整体简洁清晰，符合课程作业中适度抽象的建模要求。

------

### 8. 教师授课表

“教师—教授—课程”在原始基数上属于 1:N 联系，通常可通过在课程表中设置教师编号外键来实现。
但考虑到该联系本身具有“开课学期（比如2024-2025秋）”这一属性，因此将其单独建立为**教师授课表**，用于记录某教师在某学期开设某课程这一事实。

该表属于由带属性联系转化而来的关系表，其设计体现了当联系本身包含业务含义时，即使不是多对多联系，也可以单独抽取为表。
这样设计既保留了模型的简洁性，又体现了对关系属性的进一步分析和表达能力。

------

### 9. 活动表

**活动**用于保存校园活动的基本信息，包括活动编号、活动名称、开始时间、结束时间和主办单位。
根据“建筑—举办—活动”为 1:N 联系，在活动表中设置“建筑编号”外键，表示每项活动在哪栋建筑举办。
活动表与用户之间还存在多对多参与关系，因此活动表本身保存活动基本属性，参与关系则交由专门的关系表处理。

------

### 10. 查询记录表

**查询记录**用于记录用户的查询行为，包括查询类别、查询内容和查询时间。
根据“用户—产生—查询记录”为 1:N 联系，在查询记录表中设置“用户编号”外键，表示每条查询记录由哪个用户产生。
将查询记录单独建表，有利于完整保留历史查询信息，也便于后续统计用户行为和分析系统使用情况。

------

### 11. 用户活动参与表

“用户—参加—活动”为 **多对多联系**，不能直接在任一实体表中用单个外键表示，因此需要单独建立**用户活动参与表**。
该表以“用户编号 + 活动编号”作为联合主键，用于记录某用户参加某活动这一事实。
该表属于典型的**关系表**，其主要作用是实现多对多联系的关系模式转换，并支持后续查询“某用户参加了哪些活动”或“某活动有哪些参与用户”。
